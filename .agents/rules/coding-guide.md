# コード編集時の心構え
## 型を使う
TypeScriptの定義された型に沿って、素直にコードを書きます。  
必要であれば、自分でも型やインターフェースを定義してください。

## 既存の実装を参考に
既存の実装を探して参照したり、同じ考え方でコードを書きます。目的に合ったnpmパッケージがあればインストールして使います。

ただし、npmパッケージはメンテナンス状況を確認して導入すること。更新日の古いnpmパッケージは要注意です。

## ドキュメントを参照する
APIやライブラリのことを知るためにソースコードを読み込みたくなるのはわかりますが、まずはドキュメントや仕様書を参照したり思い出したりします。  
具体的には以下のようなものです。

- MDN
- IETF RFC
- AGENTS.md
- SKILL.md
- npmパッケージのREADMEやllms

## 1ファイルの行数を適切に
1つのファイルに500行近いコードを書くとかなりやりすぎな感じがあります。  
特に複雑もしくは行数の多いロジックは、関数にして新しいファイルに分割します。

## コメントを残そう
コメントまみれでも読みにくくなってしまいますが、次のような場面では積極的にコメントを残すようにしてください。

- 調べて初めて発見した、APIやnpmパッケージなどの仕様
- 直感に反する、トリッキーだが動作した実装
- 複雑な思考を経て修正できた箇所
- ワンライナーコードの解説

エージェントスキル `.agents/skills/*/SKILL.md` に書き残すことも考えてください。  

作業中のIssueやPRだけに関する事柄であれば、複雑だった箇所をIssue・PRコメントで残すと良いと思います。  
Issueには仕様やnpmパッケージの選択についてのまとめ、PRには具体的なコードに関することを書くと良いです。

Agent MemoryなどClaudeで準備されているメモリ機能よりは、コード内コメント、GitHubコメントやSKILL.mdを使って、人間でもアクセスしやすいようにしてください。

## 作業粒度とGit コミット
### 粒度
1コミットに対する変更の粒度は、次のようにしてください。

- 1つの機能が正常に動作するようになったら、テストがあればそれが動いたら

### CHANGELOGを書く
一番上のChanges以下に、変更内容を書きます。コミットコメントと同じようなプレフィクスを付加してください。

### コミットコメント
- `<whatToDo>(<scope>): ...`
- whatToDo: refactor, feat, enhance, perf(-ormance), chore
- scope: client, worker, api, bgzf, sw
  * 主要なスコープが1つでないなら括弧は省略
- 例: `perf(client): change build storategy of i18n from virtual to inline-chunks`

実装作業が全て完了していない場合、先頭に`[WIP] `を付加。

### push
ブランチで作業している場合はすぐにpushすること。

## PR/Issue説明文更新
PR/Issueの変更をした場合、それに合わせて説明を更新したりコメントで更新内容を書き遺したりするようにします。
